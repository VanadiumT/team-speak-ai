# 音频管线

> 定义音频从 TeamSpeak 捕获到 AI 处理后播放回 TeamSpeak 的完整链路和处理模式。

---

## 1. 音频链路全景

```
TeamSpeak 客户端
  │ 麦克风采集 (PCM 16kHz 16bit mono)
  ▼
Java Bridge (ts3j)
  │ 每 10ms 推送一帧音频
  ▼
WS :8080/teamspeak/voice
  │
  ▼
Python Backend
  ├── AudioBus (发布/订阅)
  │     ├── STT 订阅者 ──→ 语音识别
  │     └── 回环订阅者 ──→ 回声消除/监听
  │
  ▼
AI 处理链 (STT → LLM → TTS)
  │
  ▼
Java Bridge ← WS :8080/teamspeak/voice
  │ 播放队列
  ▼
TeamSpeak 客户端 (听到 AI 回复)
```

---

## 2. AudioBus — 发布/订阅模型

### 2.1 设计动机

TeamSpeak 频道内可能有多人同时说话，音频来源是单一的 WS 连接，但消费者可能有多个（STT 识别、回环监听、录音保存等）。发布/订阅模型解耦了生产者和消费者。

### 2.2 模型

```
                  ┌→ 订阅者 A (STT 识别)
AudioBus (发布) ──┼→ 订阅者 B (回环监听)
                  └→ 订阅者 C (录音)
```

- **发布**：`ws_teamspeak.py` 收到 Java 桥音频后 publish 到总线
- **订阅**：消费者通过 `subscribe(listener_id)` 获取 `asyncio.Queue`
- **取消订阅**：消费者断开时 `unsubscribe()`
- **背压处理**：队列满时丢弃最旧帧（语音场景优先实时性）

### 2.3 音频帧格式

每帧携带：原始 PCM 数据 + `sender_id`（区分说话人）。

## 3. 回环监听

### 3.1 问题

TeamSpeak 默认不把本客户端的麦克风音频送回给客户端自己，这使得 STT 无法"听到"用户说的话。

### 3.2 解决方案

启用 TeamSpeak 回环（loopback）模式，让客户端也能收到自己发出的音频。配置在 TS 输入节点中：

- `loopback: true` — 捕获本地麦克风音频
- `sample_rate: 16000` — 采样率
- `channels: 1` — 单声道

---

## 4. 音频缓冲

### 4.1 问题

音频是连续流，但 STT 识别需要句子级别的音频段。需要一个缓冲机制在接收音频帧的同时检测语音边界。

### 4.2 设计

- 按 `sender_id` 区分说话人，每人独立缓冲
- 语音活动检测 (VAD) 判断说话开始/结束
- 超时 (2s 静音) 自动切断当前缓冲段
- 超过最大缓冲大小时丢弃旧数据

---

## 5. VAD — 语音活动检测

### 5.1 作用

VAD 是一段语音前的预处理节点，将连续音频流切分为有意义的句子片段：

```
连续 PCM 流 → VAD → [分句1] [分句2] [分句3] ...
```

每个分句独立触发下游处理（如 STT → LLM → TTS），实现"说一句回一句"的实时对话。

### 5.2 流式输出

VAD 以流式模式输出分句音频，下游可以逐句开始处理而不用等整段话说完。

---

## 6. 音频格式约定

| 参数 | 值 | 说明 |
|------|-----|------|
| 采样率 | 16000 Hz | 语音识别标准采样率 |
| 位深 | 16 bit | PCM 格式 |
| 声道 | 1 (mono) | 语音场景不需要立体声 |
| 编码 | PCM (原始) / MP3 (压缩后) | 传输用 MP3 压缩，处理前解码为 PCM |
| 帧间隔 | ~10ms | Java 桥推送间隔 |

---

## 7. 音频分片防溢出

流式音频处理中，上游可能生产速度快于下游消费速度。防护策略：

- 每订阅者队列设 `maxsize`，满时丢弃最旧帧
- 丢弃时记录日志，便于监控
- 关键路径（STT）可设更大的队列缓冲
