# 架构设计

> 定义系统分层、通信模式、数据归属和生命周期。是所有设计决策的根基。

---

## 1. 核心原则

### 1.1 后端持有全部状态

```
前端 (编辑器 + 渲染器)         后端 (数据所有者 + 执行引擎)
┌──────────────────────┐      ┌──────────────────────────┐
│ 编辑器层              │      │ 数据管理层                │
│ · 拖拽节点            │◄──WS─►│ · 流程 CRUD              │
│ · 创建连线            │      │ · 坐标/配置/历史存储     │
│ · 撤销/重做           │      │ · 连线和端口校验         │
├──────────────────────┤      ├──────────────────────────┤
│ 渲染层                │      │ 执行引擎                  │
│ · 画布 + SVG连线     │      │ · 节点调度               │
│ · 节点状态动画        │      │ · 实时状态推送           │
│ · 侧栏树 / 面板       │      │ · 流式数据输出           │
└──────────────────────┘      └──────────────────────────┘
```

**三条铁律**：

1. **后端不猜前端布局** — 存储坐标但不计算布局。前端首次创建给位置，用户拖拽调整
2. **前端不存业务数据** — 所有数据由后端持有。浏览器刷新后通过 `flow.load` 恢复
3. **一切走 WebSocket** — 不混用 REST。统一信封协议，见通信设计

### 1.2 为什么选择 WebSocket 而非 REST

- 实时性：管线执行中状态变更、流式数据需要即时推送
- 双向性：前端编辑操作和后端状态推送共用同一通道
- 简单性：单一协议，不需要维护 HTTP 路由 + WS 路由两套体系
- 文件上传：Binary frame 直传，不需要 multipart 编码

## 2. 双端点设计

| 端点 | 用途 | 连接方 |
|------|------|--------|
| `/ws` | 流程管理、节点编辑、配置、执行、状态推送 | 前端 ↔ 后端 |
| `/ws/teamspeak` | 语音桥接（转发 Java 桥音频协议） | Java 桥 ↔ 后端 |

**为什么分离**：
- 音频通道数据量巨大（每 10ms 一帧），会阻塞管理消息
- 音频协议和管理协议语义完全不同，分离后各自独立演化
- 前端不直接连接 `/ws/teamspeak`，由后端内部管理

### 2.1 `/ws` 路由模块拆分

`/ws` 端点的消息由 `ws_main.py` 统一接收，按 `action` 前缀分发到领域模块：

```
ws_main.py (路由枢纽)
  ├── ws_editor.py   — 节点/连线 CRUD、撤销重做、配置更新
  ├── ws_flow.py     — 流程 CRUD、分组、导入导出、侧栏树
  ├── ws_exec.py     — 管线启停、节点触发、状态订阅
  ├── ws_presets.py  — 6 类预设 CRUD（LLM/TTS/STT/OCR/VAD/TS）
  ├── ws_teamspeak.py — TS 语音桥接中转（独立通道）
  └── ws_utils.py    — 共享工具（鉴权、错误格式化、广播）
```

拆分原则：每个模块职责单一，`ws_main.py` 只做路由分发，不包含业务逻辑。

## 3. 通信设计

### 3.1 信封模型

所有 `/ws` 消息共用统一外层结构，`action` 字段路由到具体处理器：

```
{ msg_id, flow_id, type, action, params, ts }
```

**四种消息类型**：

| type | 方向 | 语义 |
|------|------|------|
| `command` | 前 → 后 | 请求执行操作（如 node.create） |
| `event` | 后 → 前 | 状态变更推送（可能是 command 结果，也可能是主动推送） |
| `ack` | 后 → 前 | 对 command 的同步确认 |
| `error` | 后 → 前 | 错误响应 |

### 3.2 命令/事件分离

前端发 `command`，后端回 `event` + 可选 `ack`。服务端也可主动推送 `event`（如执行状态变化）。

### 3.3 flow_id 路由

一条 WebSocket 连接可操作多个流程。`flow_id` 在后端路由到对应引擎实例。切换流程不需要重连。

### 3.4 Binary Frame 文件上传

大文件不走 base64（膨胀 33%），直接走 WebSocket binary frame：
- 先发 text frame 声明元数据（`file.upload_start`）
- 后发 binary frame 传数据块
- 通过 `msg_id` 关联
- 支持断点续传

## 4. 生命周期

### 4.1 连接 → 加载 → 编辑 → 执行 → 关闭

```
connect → handshake → flow.load → [编辑循环] → [执行循环] → disconnect
                           ↑                         │
                           └── 重新加载 ─────────────┘
```

### 4.2 编辑与执行的互斥

- 流程**执行中**（status=running）：后端拒绝编辑类 command，返回 `PIPELINE_RUNNING`
- 前端收到 `pipeline.started` 时进入只读，收到 `pipeline.completed` 时恢复编辑

### 4.3 断线重连

- 前端维护心跳：每 30s 发 `ping`，90s 无 `pong` 则断开
- 重连后自动 `flow.load` 恢复完整状态
- 服务端 90s 无活动断开客户端

## 5. 服务容器 — AppContext

`core/app_context.py` 是应用全局服务容器，替代了之前分散的 12+ 个 `init_xxx()` / `get_xxx()` 全局单例：

```
AppContext.create()  →  初始化所有服务
  ├── FlowManager
  ├── HistoryManager
  ├── ConfigDefaultsManager
  ├── PresetManager (LLM)
  ├── TtsPresetManager
  ├── SttPresetManager
  ├── OcrPresetManager
  ├── VadPresetManager
  ├── TeamSpeakPresetManager
  ├── SysVarManager
  ├── ChunkReceiver
  ├── NotificationManager
  ├── PreheatConfigManager
  └── PipelineEngine
```

全应用通过 `get_app_context()` 统一访问，消除了隐式全局状态依赖。

### 5.1 模型预热

OCR/STT 等本地模型首次加载耗时 10-30 秒。预热机制在应用启动时后台加载已配置的默认模型：

```
AppContext.start_background_tasks()
  └── asyncio.create_task(preheat_models(ctx))
        ├── 读取 PreheatConfig（data/defaults/preheat.json）
        ├── 校验 OCR 默认 preset 配置完整性
        │     ├── 完整 → run_in_executor 加载模型 → 写入 _ocr_cache
        │     └── 不完整 → logger.warning 跳过
        ├── 校验 STT 默认 preset 配置完整性
        │     ├── 完整 → run_in_executor 加载模型 → 写入 _stt_cache
        │     └── 不完整 → logger.warning 跳过
        └── 完成日志
```

配置持久化在 `data/defaults/preheat.json`，前端通过"系统设置 → 模型预热"面板控制开关。

## 6. 数据存储

```
backend/data/
  flows/          # Flow JSON (管线定义)
  groups.json     # 目录/分组层级
  history/        # 每流程操作历史 JSONL (undo/redo)
  defaults/       # 每节点类型默认配置 + 预设 (llm/tts/stt/ocr/vad/ts_presets.json + preheat.json)
  uploads/        # 用户上传文件 (by upload_id)
  notifications/  # 通知历史 JSONL + read_state.json
  system_vars.json # 全局跨流程变量 (SysVarManager)
```

## 7. 安全与错误处理

### 6.1 安全

- 文件上传校验魔数（magic byte），拒绝非预期格式
- 上传文件大小限制（每节点类型可配置 `max_size_mb`）
- 不信任前端发来的所有数据，后端独立校验（端口存在性、类型兼容、无环）
- WebSocket 握手阶段校验 token（`ws_auth_token` 配置）
- 预设 API Key 下发前脱敏（`mask_presets_data`）
- LLM 输出经 DOMPurify 净化后渲染（防 XSS）
- PipelineEngine 共享状态用 `asyncio.Lock` 保护临界区
- Java Bridge `ts3Client` 字段声明 `volatile` 保证跨线程可见性

### 6.2 结构化异常体系

`core/exceptions/` 定义了领域异常层次结构，替代散落的裸 `Exception`：

```
AppException (base)
  ├── AudioException    — 音频处理错误
  ├── FlowException     — 流程管理错误
  ├── PipelineException — 管线执行错误
  ├── ProviderException — AI 提供商错误
  └── WsException       — WebSocket 协议错误
```

每个异常携带错误码和上下文信息，前端可通过错误码精确展示对应提示。`main.py` 中的全局异常处理器捕获所有未处理异常，通过日志系统记录并返回统一错误格式。

### 6.3 日志自修复

日志系统内置自修复机制：当日志 handler 发生异常时自动恢复，避免因日志故障导致服务不可用。所有标准 `logging` 通过 `LoggingHandler` 桥接到管线事件系统，保证日志在 UI 和文件中同时可见。
