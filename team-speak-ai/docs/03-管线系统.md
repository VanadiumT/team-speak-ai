# 管线系统

> 定义管线的抽象模型：节点、端口、连线、数据流、触发机制、变量系统。
> 本文档只描述**模式和约定**，不绑定具体节点类型。

---

## 1. 管线抽象模型

管线是一个有向图（DAG），由三种元素组成：

```
节点 (Node) ──连线 (Connection)──→ 节点 (Node)
    │                                    │
    └── 端口 (Port) ────────────────── 端口 (Port)
```

- **节点** — 处理单元，接收输入、执行逻辑、产生输出
- **端口** — 节点的数据进出口，定义输入/输出的类型和语义
- **连线** — 端口之间的有向连接，承载数据或事件传递

---

## 2. 端口体系

### 2.1 流式与非流式双通道

实时音频场景要求区分两种数据处理模式：

| 通道 | 前缀 | 数据特征 | 适用场景 |
|------|------|---------|---------|
| **流式** | `stream-*` | 逐片产生/消费，每片独立推送 | 实时音频、LLM 逐 token 输出 |
| **非流式** | `batch-*` | 全部就绪后一次性处理 | 图片 OCR、完整文本 |

**设计规则**：如果一种能力同时支持流式和非流式，两套端口都提供。如果只支持一种，只提供对应端口。

### 2.2 数据端口与事件端口

端口按承载内容分两类：

| 类型 | data_type | 用途 | 交互 |
|------|-----------|------|------|
| **数据端口** | `image`, `audio`, `string`, `string_array`, `messages`, `number`, `any` | 传递业务数据 | 可拖连、可配置固定值/参数引用 |
| **事件端口** | `event` | 传递控制信号（触发、完成） | 可拖连事件连线 |

**关键区别**：事件端口不传递业务数据，只传递"时刻"——触发发生、处理完成。事件连线在"事件流"视图下可见。

### 2.3 控制端口约定

所有节点统一使用以下保留端口 ID：

| 端口 ID | 方向 | 类型 | 语义 |
|---------|------|------|------|
| `trigger-in` | 输入 | event | 触发信号：收到后节点开始执行 |
| `done` | 输出 | event | 完成信号：节点执行结束后发出 |

### 2.4 端口可见性

| 可见性 | 行为 |
|--------|------|
| `always` | 始终显示 |
| `on-demand` | 仅在以下情况显示：已有连线、已配置数据来源（固定值/参数引用）、用户手动开启 |

`on-demand` 机制让节点在流程模式下保持简洁——无连接的端口自动隐藏，有数据流时才露出。

### 2.5 可重复端口 (repeatable)

某些端口需要动态增减实例（如同一节点接收多个上下文输入），通过 `repeatable` + `group` 机制：
- 端口定义为模板，`min`/`max` 控制实例数范围
- 用户在 IO 管理面板中增删实例
- 每个实例有独立 ID（如 `ctx-in1`, `ctx-in2`, ...）

## 3. 数据类型

### 3.1 基础类型

| 类型 | 说明 | 示例 |
|------|------|------|
| `string` | 文本字符串 | OCR 结果、用户输入 |
| `string_array` | 字符串数组 | 多轮对话历史 |
| `messages` | 结构化消息数组 | LLM 上下文 `[{role, content}]` |
| `image` | 图片数据 | PNG/JPG/WEBP |
| `audio` | 音频数据 | PCM 帧、WAV buffer |
| `number` | 数值 | token 计数、置信度 |
| `event` | 控制信号 | 触发、完成 |
| `any` | 泛型 | 不限制类型的通用数据 |

### 3.2 兼容规则

连线只能在兼容类型间创建：

```
image  →  image
audio  →  audio
string →  string, string_array
string_array → string_array, messages
messages → messages
number →  number
any    →  any (所有类型)
event  →  event (仅事件连线)
```

跨类型连接直接拒绝。

## 4. 触发与执行

### 4.1 触发模型 — 触发线是主宰

触发线 (trigger-in) 控制执行节奏，数据线只提供数据（payload）。核心规则：

```
[上游数据] ──→ data-in ──┐
                          │  ← 数据只是参数，不决定执行时机
[上游触发] ──→ trigger-in ┘  ← 触发才是命令，决定何时执行
                           │
                           ▼
                     节点执行 → data-out ──→ [下游]
```

| 情况 | 行为 |
|------|------|
| 有 trigger-in 连线 | **触发线主宰**。数据到达只是存着，真正执行等触发信号。触发信号一到，拿当前可用的数据执行 |
| 无 trigger-in 连线 | 降级为数据驱动——数据到达即触发执行（用于不需要精确控制时序的简单场景） |

**为什么触发线是主宰**：实时语音场景下，多个数据源（音频帧、上下文、历史记录）的到达时间不可控，但"何时处理"应由明确的触发信号决定——比如 VAD 检测到一句话结束才触发，而不是某个数据端口到了一帧就触发。

### 4.2 流式执行

流式场景下，上游逐片推送数据，节点逐片处理并输出：

```
stream-in: [分片1] [分片2] [分片3] ... [EOF]
               │       │       │
               ▼       ▼       ▼
            [处理]  [处理]  [处理]
               │       │       │
               ▼       ▼       ▼
stream-out: [分片1] [分片2] [分片3] ... [done]
```

非流式场景则是全部数据一次性输入 → 处理 → 一次性输出。

### 4.3 节点行为模式

所有节点遵循四种抽象执行模式。模式决定了节点"何时启动"和"如何运行"：

| 模式 | 说明 | 触发方式 | 典型特征 |
|------|------|---------|---------|
| **Source** | 数据起点，无上游数据线 | 手动触发、外部事件、或流程启动自动触发 | 仅有输出端口（或少量触发输入） |
| **Transform** | 数据进→处理→数据出 | 上游 trigger-in 触发，或数据就绪自动触发 | 标准 IPA：input → process → output |
| **Listener** | 后台持续运行，自循环 | 流程启动时自动开始，持续监听外部事件 | 有 `listening` 状态，条件满足时产出数据并触发下游 |
| **Sink** | 消费数据，不产出下游业务数据 | 上游 trigger-in 触发 | 输出端口仅 `done` 事件，无业务数据输出 |

这四种模式是管线引擎调度的基础。具体节点类型的端口分配和执行逻辑均遵循其所属模式。

## 5. 变量系统

### 5.1 两套变量

| 系统 | 存储位置 | 生命周期 | 用途 |
|------|---------|---------|------|
| **流程参数** | `flow.params` | 流程级，跨节点共享 | 运行时上下文（如游戏模式、语言） |
| **系统变量** | 后端全局 | 跨流程持久化 | 全局配置（如 API key、默认设置） |

### 5.2 显式读写模式

变量不通过模板字符串隐式替换，而是通过专用节点**显式读写**：

- **读节点**：从存储中取出值，输出到数据端口，可连线到下游
- **写节点**：从上游连线接收值，写入存储

**设计动机**：数据流在画布上可见可追踪。隐式 `$param.xxx` 模板替换难以调试。

### 5.3 Start 节点 — 流程入口

Start 节点是管线的**唯一入口点**。流程启动时，引擎查找 Start 节点并自动执行：

```
流程启动 → 引擎查找 Start 节点
  ├── auto_run=true → 自动执行
  ├── init_params 非空 → 逐对写入 flow.params
  ├── 发出 event-out 触发下游
  └── 发出 data-out 携带初始数据（可选）
```

Start 节点的关键配置：

| 字段 | 说明 |
|------|------|
| `auto_run` | 流程启动时是否自动执行（默认 true） |
| `init_params` | 启动时注入的初始参数 `{key: value, ...}` |

Start 属于 **Source** 行为模式——无上游数据线，由流程启动事件触发。

### 5.4 流程参数的输入来源

| 来源 | 说明 |
|------|------|
| **Start 节点 init_params** | 流程启动时注入初始参数 |
| **flow_var_write 节点** | 执行过程中动态写入 |
| **外部传入** | 前端/API 调用时携带参数 |

## 6. 数据来源配置

编辑模式下，未连线的数据输入端口可配置替代数据来源：

| 来源 | 说明 |
|------|------|
| **连线数据** (默认) | 从上游节点通过连线获取 |
| **固定值** | 用户直接输入的静态文本 |
| **流程参数** | 引用 `$param.key_name`，运行时解析 |

配置存储在 `node.config._port_sources[portId]`，有数据来源的 `on-demand` 端口在流程模式下也保持可见。
