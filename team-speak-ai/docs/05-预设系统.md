# 预设系统

> 预设系统是一种**通用配置管理机制**，用于参数多、方案多的场景下快速保存和切换配置。
> 不绑定 AI 能力。AI 只是当前的主要应用场景。

---

## 1. 设计动机

### 1.1 问题

当一个节点或能力有**大量可配置参数**时，直接内联配置有两个痛点：

- **切换成本高** — 换一个 API 提供商意味着要改 api_key、base_url、model_name 等一堆字段，容易出错
- **复用困难** — 多个节点用同一套参数时，每个节点都要复制粘贴一遍，改一处需要同步改多处

### 1.2 解决思路

将配置**分层**：把相对固定的部分（提供商、模型默认参数）提取为预设，节点只引用预设 + 少量覆写：

```
内联配置 (旧):  每个节点独立维护全部参数 → 改一处改全部
预设引用 (新):  节点引用预设 ID + 差异化的 override → 改预设即改全部
```

---

## 2. 三级模型

```
预设组 (Preset Group)
  └── 预设项 (Preset Item)
        └── 覆写 (Override)
```

这是一个通用的分层配置模式，不限定于任何具体领域：

### 2.1 预设组

一组相关配置的集合。通常对应一个"类别"或"能力"：

| 字段 | 说明 |
|------|------|
| `id` | 唯一标识 |
| `name` | 显示名称 |
| `config_fields` | 该组管理哪些配置字段 |

一个预设组下有多个预设项。

**当前系统中的预设组**：LLM、TTS、STT、OCR、VAD、TS。但预设组可以扩展到任何参数密集的节点类型。

### 2.2 预设项

一组**预保存的配置方案**，代表一种具体的配置组合：

| 字段 | 说明 |
|------|------|
| `id` | 唯一标识 |
| `name` | 方案名称（如 "MiniMax-M1"、"本地 EasyOCR"） |
| `params` | 该方案的默认参数 |

### 2.3 覆写

节点引用预设时，可对个别参数做差异化覆写：

```
节点配置 = 预设项.params + 节点.overrides
```

未覆写的字段继承预设项的默认值，覆写的字段以节点为准。

---

## 3. 使用流程

```
1. 在设置中创建预设组 → 预设项（保存一套参数方案）
2. 在节点中选择预设组 → 预设项
3. （可选）覆写个别参数
4. 需要切换方案时，节点只需换一个预设项 ID
```

**典型场景**：

- 开发环境和生产环境用不同的 API key — 切换预设项即可
- 快速对比两个模型的效果 — 复制节点，各选不同预设项
- 批量调参 — 修改预设项的默认参数，所有引用该预设的节点自动生效

---

## 4. 持久化

每个预设组独立 JSON 文件：

```
data/llm_presets.json
data/tts_presets.json
data/stt_presets.json
...
```

节点引用时只存引用 ID：

```json
{
  "preset_group_id": "llm",
  "preset_item_id": "uuid-of-item",
  "overrides": {
    "temperature": 0.7
  }
}
```

---

## 5. 前端管理

每种预设组有对应的设置面板（如 `LlmPresetsPanel.vue`），支持：

- 新建/编辑/删除预设项（含全部参数）
- 预设项列表浏览和搜索
- 节点中选择时通过下拉切换

侧栏「节点设置」分区提供各预设组的快捷入口。

---

## 6. 扩展方式

任何参数密集的节点类型都可接入预设系统：

1. 定义预设组 → 新建 `xxx_presets.json`
2. 实现后端 preset manager → 提供 CRUD + 查询接口
3. 添加前端面板 → `XxxPresetsPanel.vue`
4. 节点注册时声明使用该预设组

**不是 AI 专属，是需要"快速切换配置方案"的任何场景都可以用。**
